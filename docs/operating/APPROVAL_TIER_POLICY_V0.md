## 왜 이게 더 안전한가

- "다 승인"(현재): 사소한 일까지 오너 대기 → 자리 비우면 멈춤 + 진짜 위험한 결정이 사소한 100개에 묻혀 *대충* 통과 → **역설적으로 덜 안전.**
- "다 막 만들기": 빠르지만 스키마 파괴·비밀 유출을 못 막음.
- **정답:** 위험으로 나눠서, 되돌릴 수 있는 95%는 승인 제거 → 오너는 위험한 5%에만 집중 → **더 빠르고 더 안전.**

## 3-Tier 기준표

| Tier | 무엇 (예) | 되돌리기 | 승인 | 처리 방식 |
|---|---|---|---|---|
| 🟢 **저위험** | 문서/README 수정, 라벨 정리, 끝난 항목 archive, UI 문구, 주석, 테스트 추가, 포맷팅, 워크플로우 카탈로그 라벨 | 1줄/즉시 | **불필요** | 그린이면 self-merge. 승인 개념 자체를 둔다 |
| 🟡 **중위험** | 신규 기능 슬라이스, 워크플로우 신설/통합, 비파괴적 데이터 추가, 의존성 추가, 폴더 구조 변경 | 어렵지만 가능 | **사후 통지** | self-merge 후 변경 로그/PR에 한 줄 기록. 오너는 비동기 확인 |
| 🔴 **고위험** | DB 스키마 변경/마이그레이션, 비밀값·원문·DB의 외부 전송, public 저장소 canon 승격, 권한/접근 경계 변경, 결제·구매·송금, 대량 삭제(attic 아닌 실삭제) | 어렵거나 불가 | **사전 승인 필수** | 진행 전 오너 ack. PR 코멘트 한 줄이면 됨(패킷 문서 안 씀) |

## 라벨 운영 (머지 규칙과 연결)

PR/커밋에 다음 라벨이 **하나라도** 붙으면 🔴로 간주, 오너 사전 ack 전까지 머지 금지:

- `schema` — DB/데이터 모델 변경
- `secret` — 비밀값·credential 취급
- `external` — DB/원문/ingest의 외부 전송 (LLM 메타데이터 레인은 기존 owner 승인 경계 따름)
- `public-canon` — public GitHub 정본 승격
- `access` — 권한/접근/네트워크 경계 변경

라벨 없음 + 테스트 그린 = 🟢/🟡로 보고 self-merge 허용.

> AGENTS.md 머지 규칙 교체안과 1:1 연결:
> **"`node --test` + `npm run validate` 그린이면 self-merge. 단 `schema`/`secret`/`external`/`public-canon`/`access` 라벨이 붙은 변경만 owner가 사전 ack."**
