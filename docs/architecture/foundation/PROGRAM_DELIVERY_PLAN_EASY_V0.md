## 0. 한 문단 요약

우리는 **팀이 실제로 쓰는 ERP**를 만들려고 합니다. 그런데 "프로그램을 잘 짓고, 안전하게 열고, 회사 시스템(Soulforge)과 제대로 잇는 일"을 책임질 **전담 감독**이 없습니다. 이 문서는 그 감독 역할을 하나 만들고[전문가 서브에이전트], ERP가 Soulforge와 어떻게 연결되는지 규칙을 정하고[연결 아키텍처], "팀에 열기 전 반드시 통과할 안전 검문소"를 정해서 — 이 셋을 ERP 만드는 일과 **같이** 준비하자는 계획입니다.

비유로 말하면: ERP는 짓고 있는 **건물**이고, 이 문서는 **현장 감독관**을 한 명 세우고, 그 건물이 회사 본사 시스템과 어떻게 연결될지 배관·전기 도면을 그리고, **준공검사 9가지**를 정하는 일입니다.

## 1. 왜 지금 이게 필요한가 (배경)

지금 상태를 솔직히 보면:

- ERP 앱[dev-erp]은 **이미 진짜로 돌아갑니다.** 데이터 저장도 되고, 누가 뭘 했는지 기록도 남고, 실제 회사 데이터(과제 13개·메일 675건)도 들어가 있습니다.
- 그런데 **팀에게 열기엔 빠진 게 있습니다.** 로그인이 없고(아무나 접속하면 다 보임), 통신이 암호화 안 됐고, "누가 무엇을 볼 수 있나" 권한이 강제되지 않고, 자동 백업이 없고, 개인정보(근태 등) 다룰 규칙이 없습니다.
- 더 큰 문제: 건물을 **짓는 사람**[dev_worker]은 있는데, "무엇을 어떤 순서로, 어떤 검사를 통과해야 문을 열지"를 잡아주는 **감독**이 없습니다.

한 줄: **"팀이 쓸 물건을 안전하게 완성하고 본사 시스템과 잇는 책임자"가 비어 있고, 그게 곧 ERP 오픈 위험입니다.**

## 2. 이 계획이 스스로 지키는 약속 (원칙)

1. **최소한만 새로 만든다.** 이미 있는 역할·도구·연결점을 최대한 재활용한다.
2. **중복 안 만든다.** 이미 있는 "짓는 사람"이나 다른 보조 역할과 같은 일을 두 번 정의하지 않는다.
3. **ERP는 원본이 아니라 거울이다.** 진짜 데이터의 주인은 Soulforge고, ERP는 그걸 보여주고 굴리는 화면일 뿐. ERP가 "또 하나의 원본"이 되면 안 된다[파생 소비자 불변식].
4. **공개/비공개를 섞지 않는다.** 코드·구조는 공개 저장소에, 실제 업무 데이터·개인정보는 비공개에. 비밀번호 파일은 AI가 열어보지 않는다.
5. **최종 결정은 사람(owner)이 한다.** AI는 준비·정리·검증·경고까지. 실제 "열어도 된다"는 승인, 합치기, 배포, 개인정보 수집 결정은 사람이.
6. **한 번 됐다고 정답으로 굳히지 않는다.** 여러 번 검증된 방법만 회사 표준으로 등록한다[승급].

## 3. "전문가 감독"을 만든다 (서브에이전트)

### 3.1 이 감독이 하는 일 (한 줄)

> 프로그램(만들 물건 단위)을 **정의하고 → 짓는 순서를 짜고 → 팀에 열어도 되는지 검사하고 → 인도하고**, 잘 된 방법은 회사 표준으로 등록하는 **현장 감독관**. 직접 망치질(코딩)·합치기·배포는 안 하고, 실제 작업은 "짓는 사람"[dev_worker]에게 작업지시서로 넘긴다.

게임식 이름으로는 **도편수(현장 총괄 목수)**. 회사의 "사장"[길마]이 일을 맡기면, 그 일을 끝까지 책임지고 짓는 감독입니다.

### 3.2 다른 역할과 안 겹치게 (경계)

| 역할 | 위치 | 하는 일 |
| --- | --- | --- |
| 사장[guild_master] | 맨 위 | 무슨 일을 받을지, 누구에게 맡길지 결정 |
| **현장 감독(이 문서)** | 중간 | 무엇을·어떤 순서로·어떤 검사로 지을지 잡고, 작업지시서를 내림 |
| 짓는 사람[dev_worker] | 실행 | 작업지시서 1장을 받아 실제로 코드 짜고 검증 |
| 설계 보조[SE assistant] | 보조 | 설계 문서 준비를 돕는 하위 도우미 |
| 검사관[auditor] | 검증 | "열어도 되나" 안전 검사 |

핵심: 이 감독은 **새 작업 인력을 또 만드는 게 아니라**, 사장과 작업자 사이의 **빈 중간 자리**를 채웁니다.

### 3.3 회사 조직도에 어떻게 넣나 (정본 표현)

가장 가볍게, **새 직군을 만들지 않고 기존 3개 직군을 묶어** 팀을 하나 만듭니다[party].

- 팀 이름: `program_delivery_cell` (프로그램 인도 셀)
- 묶는 직군 3개:
  - **총관**[administrator] — 검사·경계·위임·표준 등록 담당
  - **집행관**[marshal] — 짓는 순서·마감 관리
  - **감식관**[auditor] — 안전 검사·인도 준비 확인
  - (선택) 빌드 정면 담당[knight] 또는 안정화 담당[healer]을 일 성격에 따라 추가
- 담당자: `program_delivery_lead_01` (이 팀의 리더)
- **옵션:** 아예 새 직군 "도편수"[shipwright]를 만들 수도 있지만, 기존 3개 묶음으로 충분하면 안 만드는 게 원칙(사람 결정사항 ①).

### 3.4 실제로 어떻게 작동하게 하나 (실행 표현)

이 감독은 Codex와 Claude **양쪽에서 똑같이** 작동합니다. (앞서 `CLAUDE.md = @AGENTS.md`를 깔아둬서 두 도구가 같은 지침서를 읽습니다.)

- **Claude 쪽:** `.claude/agents/program-delivery-architect.md` 라는 정의 파일
- **Codex 쪽:** `.registry/skills/program_delivery_architect/` 라는 스킬 폴더
- 둘은 같은 팀[program_delivery_cell]의 "작동 방식"이고, "누구이고 무슨 권한인가"(조직도)와 분리해서 둡니다.

### 3.5 무엇을 받고 / 내고 / 못 하나 (계약)

받는 것: 일의 목적·제약·사용자·기한, 대상 물건과 코드 이름, 현재 상태.

내는 것: 실행 계획[mission], 작업지시서[dev_worker_request], **안전 검사 보고서**(아래 9개 검문소), 표준 등록 후보.

못 하는 것(사람 권한): 본문[main]에 직접 합치기, 자동 합치기, 배포, 비밀번호 열람, 업무 원문을 외부 AI에 보내기, 개인정보 수집 결정, 모르는 값을 추측으로 채우기(→ 대신 "사람에게 물어볼 질문"으로 남김).

### 3.6 일하는 순서 (루프)

```
일 접수 → 범위 정하기 → 안전검사 계획 → 작업지시서로 쪼개기
→ 짓는 사람에게 넘기기 → 검사 → 사람에게 "열까요?" 보고 → 잘된 건 표준 등록
```

## 4. ERP를 Soulforge와 어떻게 잇나 (연결)

### 4.1 가장 중요한 규칙

ERP는 **거울**입니다. 진짜 데이터의 주인은 Soulforge이고, ERP는 그걸 비춰서 보여주고 업무를 굴립니다. ERP 안에서 생긴 기록 중 회사 자산이 될 부분만 **정해진 통로**로 Soulforge에 올라갑니다. ERP가 멋대로 "원본"이 되면 안 됩니다.

### 4.2 데이터가 흐르는 방향

- **읽기:** Soulforge(원본) → 중간 정리본[projection] → ERP 화면. ERP는 정리본을 읽기만 합니다.
- **쓰기:** ERP에서 생긴 "누가·언제·무엇" 기록 → 비공개 기록 창고[`_workmeta`]에 요약·포인터로 저장(원본 파일째로가 아니라).
- **외부 AI 경계:** 요약·메타만 보내고, 원본·DB는 절대 밖으로 안 나갑니다.

### 4.3 무엇을 어디에 두나

| 무엇 | 어디에 | 공개? |
| --- | --- | --- |
| ERP 코드·구조·가짜 샘플 | 앱 폴더 | 공개 |
| ERP 실제 DB·실데이터 | 그 PC 안에만 | 비공개 |
| ERP 과제 메타·기록 | 비공개 창고 `_workmeta` | 비공개 |
| 실제 업무 원문·첨부 | 작업장 `_workspaces` | 비공개 |
| 연결 방향 설명 문서 | `docs/architecture` | 공개 |

### 4.4 ERP를 회사 프로젝트로 등록하는 법

1. 프로젝트 코드 이름 정하기(예: `dev_erp`) — 사람 결정 ③
2. ERP 인도 계획서[mission] 1개 만들기
3. 비공개 창고[`_workmeta`]에 메타·작업큐·보고서 자리 만들기
4. 작업장[`_workspaces`]에 실데이터 두기(그 PC 안에서만)
5. 앱 코드는 지금 위치 유지, 연결은 "읽기 정리본 + 기록 요약"으로만

### 4.5 잘된 방법을 표준으로 (승급)

ERP를 굴리다 **여러 번 검증된 방법**(예: 메일→할일 전환 규칙, 백업·복구 절차)은 회사 표준[workflow/skill]으로 등록합니다. 이게 Soulforge가 원래 하려던 "사람이 한 번 잘한 일을 재사용 자산으로 바꾼다"가 ERP에서 실제로 도는 지점입니다. **한 번 됐다고 바로 굳히지 않고**, 반복·검증 후 등록합니다.

## 5. 팀에 열기 전 통과할 안전 검문소 9개 (G1~G9)

앞에서 말한 ERP 차단요건들을 **검사관[auditor]이 문 열기 전 확인하는 9개 검문소**로 정리합니다. 하나라도 막히면(사람이 예외 서명하지 않는 한) "팀 오픈"을 막습니다.

| # | 검문소 | 통과 기준 | 지금 |
| --- | --- | --- | --- |
| G1 | 로그인 | 계정·로그인 없이는 접속 불가 | 없음 |
| G2 | 통신 암호화 | 네트워크로 열면 HTTPS, 평문 금지 | 없음 |
| G3 | 권한 | 역할별로 볼/고칠 권한 강제 | 자리만 |
| G4 | 백업·복구 | 자동 백업 + **복구 1회 실제 성공** | 없음 |
| G5 | 행위자 기록 | 모든 변경이 진짜 사용자에게 귀속 | 일부 |
| G6 | 개인정보 | 근태 등 모으기 전 동의·보관·접근 규칙 | 없음 |
| G7 | 데이터 이전 | 업데이트해도 팀 데이터 안 깨짐 | 버전만 |
| G8 | 상시 가동 | 꺼지면 자동 재시작·감시되는 서버 | 임시 |
| G9 | 사용 안내 | 비개발자 안내서 + 문제 신고 통로 | 없음 |

**G1·G2·G4는 어떤 오픈이든 무조건 먼저. G6은 개인정보 기능 켜는 순간 무조건 먼저.**

## 6. 단계별 진행 (ERP 만들기와 같이)

새 기능을 막 늘리지 말고, **검문소를 닫는 순서**로 갑니다.

- **0단계 — 준비:** 감독 팀과 연결 규칙을 시스템에 등록, ERP를 프로젝트로 등록, 검문소 9개 명문화. (사람 결정 ①②③ 필요)
- **1단계 — 읽기전용 시범:** G1·G2·G4 먼저 닫고, 2~3명에게 **읽기 전용**으로 열어 실제로 써보기.
- **2단계 — 쓰기 시범:** G3·G5(권한·행위자) 닫고 쓰기 권한 단계적 개방.
- **3단계 — 개인정보:** G6(개인정보 규칙) 확정 **후에만** 근태·투입 기능 켜기.
- **4단계 — 표준화:** G7·G8·G9 안정화하고, 잘된 절차를 표준으로 등록.

각 단계마다 목적·산출물·완료기준·사람결정·검증을 둡니다(정식 기획안 6절에 상세).

- **혼자→팀 전환 비용** → 인수인계 체크포인트와 절차문서 강제.
