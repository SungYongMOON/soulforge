# Agent Fantasy — 비전 · Phase · 세계관 정리 초안

## 1) 한 줄 정의
Agent Fantasy는 메일, 일정, 프로젝트를 **던전·몬스터·보스·파티**로 변환해  
사람과 AI의 협업을 **전투**, **성장**, **로그**, **승급**의 언어로 다루는 업무 RPG 에이전트다.

## 2) 왜 만드는가
- 업무는 원래 피로를 만들기 쉽다.
- 하지만 실제 일의 흐름을 게임처럼 보이게 만들면, 우선순위와 난이도와 자동화 수준을 직관적으로 다룰 수 있다.
- 이 시스템의 목적은 “업무를 가볍게 보이게 하는 장식”이 아니라,  
  **실제 일의 구조를 더 잘 보이게 하고, AI와 사람의 역할 분담을 축적 가능한 자산으로 만드는 것**이다.

## 3) 제품 비전
### 북극성
업무를 RPG처럼 다루되, 결과는 장난감이 아니라 **실행력·자동화율·학습 자산**으로 남게 만든다.

### 핵심 약속
1. 들어오는 일은 자동으로 읽히고 분류된다.  
2. 일은 프로젝트/마일스톤/업무 난이도에 따라 몬스터나 보스로 매핑된다.  
3. 사람과 AI의 수행 방식은 전투 로그로 기록된다.  
4. 반복 성공은 자동화 숙련도와 승급으로 반영된다.  
5. 전체 프로젝트가 끝나면 “AI가 한 일 / 사람이 한 일 / 효율 변화”가 분석 가능해야 한다.

## 4) 세계관 고정안
- 메일함 = 세계의 관문 / 퀘스트 유입 지점
- 프로젝트 = 던전
- 시스템 엔지니어링 기반 대마일스톤 = 스테이지 / 층 / 챕터
- 개별 업무 = 몬스터
- 최종 납품/승인/완료 = 보스전
- 반복적으로 성공한 자동화 = 숙련 스킬
- 자동 실행 횟수와 성공률 = 경험치 / 숙련도 / 승급 조건
- 사람/AI/도구 조합 = 유닛
- 유닛 조합 = 파티
- 특정 미션용 조합 시너지 = 연계기
- 지식창고 = 도서관 / 아카이브
- 도구창고 = 무기고
- 스킬창고 = 스킬서 보관소
- 운영자 = 길드 마스터 또는 던전 관리자
- 사용자의 하루 작업 화면 = 전장 / 작전판

## 5) 현재 단계 판정
현재는 **비전·세계관·도메인 매핑은 강하지만, 실제 플레이 가능한 핵심 루프는 아직 얇은 상태**다.

### 단계 해석
- 콘셉트 성숙도: 높음
- 도메인 모델(업무→게임 매핑): 중상
- 핵심 루프(읽기→분류→배치→처리→로그→회고): 중하
- 자동화 안정성: 낮음~중하
- 관측/분석: 낮음
- 사용자 매뉴얼/온보딩: 낮음

### 현재 위치 한 줄
**“풍부한 Pre-Alpha, 아직 얇은 Vertical Slice”**

## 6) 지금 가장 먼저 해야 할 일
모든 축을 동시에 만들지 말고, **하나의 플레이 가능한 수직 단면**만 먼저 만든다.

## 7) 추천 Phase
### Phase 0 — 세계관 최소 고정
목표: 더 이상 이름과 개념을 흔들지 않기
- 20개 이하 핵심 용어 사전 확정
- 매핑표 확정: 메일/프로젝트/마일스톤/태스크/로그/자동화/승급
- 버리는 것 명시: 아직 안 만드는 기능 목록

완료 기준:
- 새 아이디어가 생겨도 기존 용어를 갈아엎지 않는다.
- 누구에게 설명해도 3분 안에 그림이 그려진다.

### Phase 1 — 실제 사용 가능한 첫 버전
목표: 매일 직접 쓸 수 있는 최소 루프 만들기
- 메일 또는 일정에서 업무 후보 수집
- 업무를 프로젝트/마일스톤별로 분류
- 각 업무를 몬스터로 등급화
- 오늘의 전장에 배치
- 처리 결과를 전투 로그로 남김

이번 단계에서 **넣지 않는 것**
- 복잡한 유닛 종족 시스템
- 정교한 경제 시스템
- 과도한 UI 연출
- 완전 자동 전투

완료 기준:
- 하루 업무를 이 시스템으로 1회 이상 끝까지 운용 가능
- 업무 누락 없이 “오늘 무엇을 해야 하는지” 보인다
- 최소한 수동+반자동으로 전투 로그가 남는다

### Phase 2 — 자동화 승급
목표: 반복 성공한 업무를 자동화 자산으로 만들기
- 동일 유형 업무 묶기
- 자동 수행률/수동 개입 횟수 기록
- 3회, 5회, 10회 연속 성공 같은 승급 규칙 도입
- “슬라임은 완전자동”, “오크는 승인 후 자동”처럼 규칙화

완료 기준:
- 업무마다 자동화 단계가 보인다
- 어떤 업무를 다음 자동화 대상으로 삼을지 판단 가능하다

### Phase 3 — 파티/유닛/지식창고 연결
목표: 사람+AI+도구의 조합을 자산화
- 유닛 속성 도입: 직업, 지식, 도구, 역할
- 파티 템플릿 도입
- 연계기 규칙 도입
- NotebookLM/MCP/외부 지식 연결

완료 기준:
- 특정 미션에 잘 맞는 파티 조합이 추천된다
- 도구/지식/프롬프트가 재사용 자산이 된다

### Phase 4 — 평가와 운영체계
목표: 프로젝트 종료 후 성과를 측정
- AI 수행량 / 인간 수행량 / 협업 비율
- 자동화 성공률 / 실패 원인 / 개입 포인트
- 프로젝트별 전투 히스토리와 회고
- 파티 효율, 직업별 강점, 병목 분석

완료 기준:
- “이 시스템이 실제로 얼마나 유용했는가?”를 데이터로 설명 가능

## 10) 사용자 매뉴얼 구조
### A. 플레이어 매뉴얼
- 이 앱은 무엇인가
- 오늘의 일은 어디서 들어오는가
- 몬스터 등급은 어떻게 해석하는가
- 오늘 해야 할 일은 어디서 보는가
- 전투를 완료하면 무엇을 기록하는가
- 승급은 어떻게 일어나는가

### B. 운영자 매뉴얼
- 던전 생성 규칙
- 마일스톤 설계 규칙
- 몬스터 분류 규칙
- 자동화 승급 규칙
- 로그 검토 규칙
- 파티/유닛 템플릿 운영 규칙

### C. 설계자 매뉴얼
- 데이터 모델
- 이벤트 스키마
- 자동화 아키텍처
- MCP/지식 연동
- 보안/권한
- 변경 이력

## 11) 세계관 문서 운영 규칙
문서는 세 종류만 운영한다.

1. **Canon**
   - 확정된 설정만
   - 용어, 세계 구조, 규칙, 승급 조건

2. **Idea Backlog**
   - 떠오른 아이디어 보관소
   - 아직 확정 안 된 설정
   - 실험 후보

3. **Decision Log**
   - 언제 무엇을 왜 채택/폐기했는지
   - 갈아엎기 피로를 막는 기록

## 13) 지금 바로 실행할 7개
1. 세계관 핵심 용어 20개를 확정한다.
2. Phase 1에서 안 만들 것 목록을 적는다.
3. “메일→몬스터→오늘의 전장→전투 로그” 한 줄만 구현한다.
4. 전투 로그 스키마를 먼저 고정한다.
5. GitHub Projects 같은 기존 보드를 임시 UI로 쓴다.
6. 사용자 매뉴얼은 플레이어용 1장부터 쓴다.
7. 1주일치 업무를 직접 돌려 본 뒤에만 다음 기능을 추가한다.

## 14) 최종 판단
전체 비전은 크다.  
하지만 **실사용 가능한 첫 버전**은 의외로 가깝다.  
멀어 보이는 이유는 기능이 너무 많아서가 아니라, **첫 번째 살아있는 루프가 아직 분리되지 않았기 때문**이다.

따라서 지금의 정답은:
- 더 크게 설계하기보다,
- 더 예쁘게 만들기보다,
- **더 좁게 잘라서 먼저 살아보는 것**이다.
