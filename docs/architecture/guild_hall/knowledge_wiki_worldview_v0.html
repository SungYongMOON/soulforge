Soulforge Knowledge Wiki

파일 창고가 아니라, 검토 가능한 지식 운영 체계

자료를 모으고, 출처를 확인하고, private wiki projection을 만든 뒤, 검토를 통과한 것만 정본 지식으로 승격한다.

왜 이런 방식인가

출처가 먼저

원본이 어디서 왔는지 모르면 나중에 LLM이 아무리 그럴듯하게 말해도 믿을 수 없다.

토큰을 아낀다

PDF를 매번 통째로 열지 않고, private projection과 index로 다시 찾을 수 있게 만든다.

바로 정본화하지 않는다

LLM 요약은 후보일 뿐이다. source와 review gate를 지나야 공유 지식이 된다.

세계관 한 장

1 Source

원본 자료

PDF, 공식 문서, 승인된 로컬 파일, 압축 해제된 ZIP 내용물.

2 Projection

비공개 위키화

source별 markdown, index, log, 질문별 흡수 노트를 만든다.

3 Candidate

지식 후보

출처, 범위, 부족한 근거를 가진 작은 후보로 쪼갠다.

4 Review

검토

claim ceiling을 정하고 public/private 경계를 확인한다.

5 Canon

정본 지식

검토를 통과한 항목만 반복 사용 가능한 공유 지식이 된다.

공간 지도

Soulforge wiki는 하나의 폴더나 하나의 앱에만 있지 않다. 역할이 다른 여러 공간을 연결해서 쓴다.

입구 로컬 PC

Desktop, Downloads, 작업 폴더에 사용자가 받은 원본 파일이 먼저 놓인다. 필요한 파일만 source intake로 넘긴다.

비공개 작업장 _workmeta

source packet, private projection, lint, concept candidate, review evidence가 쌓이는 실제 카파시식 데이터 작업장이다.

중앙 보관 Google Drive

처음 받은 후보, owner-held source, 작업 bundle, 정본 package를 모두 보관하는 자료 금고이자 백업 장소다.

정본 질의 NotebookLM

정본 package를 올려 먼저 질문하는 source-grounded 인터페이스다. 내려받을 수 있어도 주 역할은 보관소가 아니라 호출면이다.

보기 전용 Obsidian

정본 wiki entry만 보여주는 로컬 markdown view다. 후보, raw evidence, private run payload는 보이지 않게 한다.

실행 규칙 .workflow

어떤 절차로 처리할지 저장한다. PDF 본문이 아니라 workflow, template, boundary를 가진다.

상위 체인 .party

knowledge_wiki_cell처럼 여러 workflow를 순서대로 묶는다. 각 단계 실행은 subagent가 맡는다.

정본 .registry/knowledge

검토를 통과한 공유 지식만 들어간다. private projection이나 LLM 요약이 자동으로 들어오지 않는다.

사용 흔적 Access Ledger

어떤 ref가 작업에 쓰였는지 metadata만 남긴다. 사용 기록은 유용성 신호이지 진실성 증명은 아니다.

자료는 이렇게 이동한다

기본 경로는 Local에서 시작해 Drive inbox에 먼저 맡겨두고, 필요한 파일만 _workmeta에서 카파시식 projection으로 만든 뒤, review를 거쳐 Drive에 다시 백업하고 정본 package를 NotebookLM과 Obsidian view로 배포한다.

Local

받은 파일, 임시 원본, ZIP

Drive Inbox

처음 받은 후보와 원본을 보관

Intake

승인, 압축 해제, inventory

_workmeta

source packet과 private evidence

Projection

_workmeta 안에서 markdown index, log, 후보 생성

Review

claim ceiling과 경계 판단

Drive Backup

작업 bundle과 정본 package 보관

Canon

승인된 정본 지식만 등록

NotebookLM

정본 package를 먼저 질의

Obsidian

정본 wiki만 읽기 전용 표시

Drive는 무엇을 보관하나

정본만 보관하지 않는다. 처음 들어온 후보, 원본 source, 작업 bundle, review된 private 결과, canon package를 함께 보관하되 inbox, candidate, working, canon, blocked 상태를 manifest로 나눈다.

NotebookLM은 어떻게 쓰나

review를 거쳐 정본 package로 묶인 자료를 올려서 먼저 질문한다. NotebookLM은 정본을 편하게 호출하는 인터페이스이고, 백업 저장소나 정본 승인자가 아니다.

Obsidian은 폴더가 필요한가

필요하다. 다만 작업장 폴더가 아니라 canon-only export 폴더여야 한다. Obsidian은 그 폴더를 vault로 열고, 후보나 raw source는 보지 않게 한다.

공간 규칙

자료 파일의 장기 보관과 백업은 Google Drive가 맡고, 실제 카파시식 데이터 작업 위치는 _workmeta다. NotebookLM은 정본 질의 인터페이스, Obsidian은 정본 wiki만 보는 읽기 전용 vault로 둔다.

운영 규칙

규칙 의미
ZIP은 풀어서 쓴다 압축파일 자체가 아니라 해제본과 inventory를 source ref로 쓴다.
원본과 projection을 분리한다 원본의 진실성은 source packet이나 owner-held source가 가진다.
LLM 답변은 권한이 아니다 LLM이나 NotebookLM은 탐색을 도울 수 있지만 정본 승인을 할 수 없다.
claim ceiling을 붙인다 발견, 출처지원, private 검증, 정본 후보, 정본 등록을 구분한다.
public/private를 나눈다 private projection과 raw source는 공개 정본 문서에 섞지 않는다.

지금 만들어진 체인

knowledge_wiki_cell party는 wiki화를 위한 상위 실행 체인이다. party는 흐름만 기억하고, 각 workflow의 내부 실행은 fresh subagent가 맡는다. Drive 보관은 별도 승인 단계가 아니라 파일 수명주기 manifest로 다룬다. codex_skill_auto_sync가 선언되면 승인된 Codex skill이나 Google Drive connector가 매 파일마다 다시 묻지 않고 업로드/동기화할 수 있다.

순서 Workflow 역할
1 official_source_packet_collect_v0 자료와 출처 상태 확인
2 sourcebound_knowledge_packet_operating_loop_v0 카파시식 private projection과 후보화
3 knowledge_access_event_capture_v0 사용 기록과 연결 신호 정리
4 post_development_review_gate_v0 경계, claim ceiling, 승격 가능성 검토
보조 archive manifest 후보/source는 intake에, working/canon package는 projection/pipeline에 기록. sync 권한은 보관 권한이지 정본 승인 권한은 아님

현재 개발 상태

완료

  • 지식 운영 모델 문서화
  • metadata-only knowledge access ledger helper
  • sourcebound knowledge packet workflow
  • knowledge access event capture workflow
  • post-development review gate
  • knowledge_wiki_cell party 등록과 smoke run 통과

아직 아님

  • PDF 내용이 자동 정본 지식이 되는 상태
  • SE 자료 전체의 실제 wiki projection 완료
  • private projection의 public wiki 승격
  • NotebookLM이나 LLM 답변의 정본 승인

다음 단계

다음 목표는 SE 자료를 knowledge_wiki_cell로 실제 실행하는 것이다.

  • SE source packet을 정리한다.
  • PDF 내부 내용을 private projection으로 만든다.
  • page/source-cited concept candidate를 뽑는다.
  • figure, table, formula처럼 약한 부분을 gap으로 표시한다.
  • review gate에서 claim ceiling을 정한다.
  • 반복 사용 가치가 있는 것만 정본 후보로 올린다.

팀원이 기억할 한 문장

Soulforge wiki는 "자료를 쌓는 곳"이 아니라 "자료를 검토 가능한 지식 후보로 바꾸고, 필요한 것만 정본으로 승격하는 운영 방식"이다.
