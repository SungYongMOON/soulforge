## LLM 레인 데이터 경계 (2026-06-13 owner 확정 — 갈림길① 해소)

- 1차 백엔드 = **Codex CLI(구독) 헤드리스 호출(로컬 브릿지)**. 어댑터는
  전환형 — 백엔드 인터페이스(codex|local|none)로 추상화, 나중에 사내
  tool_pc 로컬 모델로 교체 가능해야 한다.
- LLM 에 보내는 컨텍스트 = **메타/요약만**. 메일 원문·민감 필드는 절대
  보내지 않는다. 컨텍스트 빌더가 **필드 화이트리스트를 코드로 강제**하고,
  모든 호출은 event_log(kind=llm_call, used_refs 에 백엔드/스킬 ref,
  data_label)에 기록한다.
- owner 인지·승인 사항: 그 메타/요약은 Codex 경유로 OpenAI 에 전송된다.
- 그 외 전부 기존 가드레일 유지: DB·원문·ingest 는 외부 전송 0, 메타만,
  localhost. "외부 전송 0"의 예외는 이 LLM 레인 하나뿐이며 범위는
  화이트리스트된 메타/요약으로 한정.
- 주입 방어: 메일 제목 등 외부 유래 메타가 컨텍스트에 들어가므로, 어댑터
  프롬프트에서 컨텍스트는 항상 "데이터"로 구획(지시문 취급 금지)한다.
- 기존 자산 앵커: guild_hall/{doctor,validate,daily_ledger}/ 실존,
  후보큐에 `agent_boot_digest_v0`(proposed, owner 보류 결정 1건 내장),
  `doctor_platform_binary_check_v0`(proposed),
  `daily_work_ledger_automation_v0`(**completed** — 업무일지 데이터 레인 존재),
  `candidate_queue_archive_policy_v0`(proposed).

규모 기준: 소=반나절 내, 중=1~1.5일, 대=2일+. (강한 모델 윈도우 기준)
