- 목적: owner 요청 — "실제 팀을 운영하는 입장에서 필요해 보이는 위젯을 다 정리". 홈 대시보드 위젯 서랍에 채울 후보 풀(pool)을 도메인별로 카탈로그화.
- 근거: ① 현행 dev-erp 데이터 모델/API ② ECount 대시보드 라이브 관찰(`_workspaces/system/사내ERP_분석/ECOUNT_대시보드개인화_관찰.md`) ③ 개발팀장/PM/SE 운영 니즈.
- 표기: **현행** = 지금 데이터/API로 바로 구현 가능 / **P2** = 약간의 집계·필드 추가 / **P3** = 신규 데이터 소스·모듈 전제. 우선순위는 "팀 운영 가치 × 구현 난이도" 직관 등급.

현재 보드 기본 위젯 5종: `projects(진행 과제)`, `today(오늘·지난 마감)`, `blocked(차단)`, `mail(최근 메일)`, `events(최근 변경)`.

## 1. 운영(group_work) — 진행 상황·내 일 중심

데이터 출처가 이미 있는 영역(summary/items/guide/events)이라 대부분 바로 구현 가능. 팀장이 매일 처음 보는 "오늘 뭐가 급한가"를 담는 핵심 그룹.

| 위젯(업무 / 판타지) | 용도 | 데이터 | 팀운영 가치 | 단계 |
|---|---|---|---|---|
| 진행 과제 현황 / 던전 현황 | 활성 프로젝트별 남은 일·마감·차단·최근 움직임 한눈 (현행 projects) | /api/summary | 최상 — 콕핏의 중심 | 현행✓ |
| 오늘·지난 마감 / 둠클락 임박 | 오늘+연체 할 일 (현행 today) | /api/items?due=soon | 최상 | 현행✓ |
| 차단 목록 / 봉인 목록 | blocked 상태 할 일 + 사유 (현행 blocked) | /api/items?status=blocked | 높음 — 병목 즉시 포착 | 현행✓ |
| 내 담당 할 일 / 내 몬스터 | 로그인 사용자 assignee 기준 필터 | /api/items (assignee) | 높음 — 개인 To-Do | 현행✓(계정 전제 시 P2) |
| 단계 게이트 대기 / 보스전 대기 | 게이트 통과 대기 중인 단계·산출물 체크리스트 | /api/guide | 높음 — 승인 병목 | P2 |
| KPI 스트립 / 전황 요약 | 차단·연체·오늘마감·미분류메일 수치 카드 (현행 홈 상단 KPI를 위젯화) | /api/summary | 중 — 요약 지표 | 현행✓ |

## 추천 우선순위 (반영 시 제안 — 결정은 owner)

1. **즉시 후보(현행 데이터로 바로):** 내 담당 할 일, 미분류 메일함, 미배정 작업, KPI 스트립, 최근 산출물. → 카탈로그를 5종에서 ~10종으로 우선 확장하기 좋음.
2. **단계 게이트 대기 / 산출물 진행률 / 팀원별 부하:** P2(집계·계정). 팀장 가치 최상이라 그다음 순위 권장.
3. **자재·구매·결재·온보딩:** P3 모듈 신설과 함께. 모듈이 생길 때 위젯도 같이.

## 설계 메모

- 위젯 = "대시보드 요약 → 클릭 시 상세 화면 점프(허브)" 원칙 유지(ECount 관찰과 동일). 팝아웃은 큰 보기, 행 클릭은 해당 화면 이동.
- 작성(✎)은 생성 가능한 위젯(할 일·요청·구매요청 등)에만 노출. 읽기 전용(메일·이력·공지)엔 미노출(ECount도 연필을 상호작용 위젯에만 노출).
- 권한: 모듈/위젯 가시성은 RBAC(menu_visibility=show, access=deny "보이지만 잠김")로 — 계정 시스템(P2b) 전제.
- fantasy 라벨은 SHARED_GLOSSARY 대조를 따르고 lexicon parity 테스트로 강제.

## 6. 대분류(상단 업무군) 체계 및 포함 범위

owner 질문: "위젯이 다 필요한데 실제 대분류를 어디로 잡을지, 어디까지 넣을지" → 위젯의 소속 대분류 기준과, 지금 어디까지 열지(범위)를 정리.

### 6.1 대분류 기준 — "다루는 대상(객체)"별 (owner 결정 2026-06-13)

기존 운영/기록·이력/자재·구매/지식·도구/팀은 ECount식 일반 ERP에서 빌려온 틀이라 이 팀 업무에서 자연스럽게 나온 축이 아니었음(owner 지적). **새 기준: 우리가 실제로 다루는 "대상(객체)"별로 대분류를 잡는다.** 각 칸이 눈에 보이는 실체라 "이게 어디 속하지"가 명확하고, 위젯도 같은 객체 기준으로 정착한다.

**대분류 6개(객체축):**

| # | 대분류 | 무엇 | 현 IA 매핑 |
|---|---|---|---|
| 1 | **프로젝트(과제)** | 과제·단계·게이트·진행 현황 | 운영 일부 |
| 2 | **할 일** | 작업/태스크·담당·마감 | 운영 일부 |
| 3 | **산출물·문서** | 납품물(SSRS 등)·보고서·회의록·표준 | 기록·이력+지식 일부 |
| 4 | **메일·소통** | 메일·결재·쪽지·공지·알림 | 기록·이력 일부 |
| 5 | **자재·거래처** | 구매·발주·재고·BOM·거래처 | 자재·구매 |
| 6 | **사람·팀** | 담당자·부하·요청·분석·연락처·온보딩 | 팀+지식 일부 |

가로지르기(KPI·캘린더·활동피드 등)는 별도 "공통" 칸을 만들지 않고 **가장 관련 깊은 객체에 귀속**(아래 6.3). 순수 유틸(계산기 등)은 객체축에 안 맞으므로 대분류로 올리지 않고 후순위 도구로 둔다.

### 6.2 어디까지 넣을지 — 범위(단계)

데이터가 있는 대분류부터 연다. 빈 대분류는 탭만 두지 말고 **데이터/모듈 생길 때 노출**(빈 칸 방지).

| 대분류 | 지금? | 근거 |
|---|---|---|
| 프로젝트 | **지금(full)** | summary 데이터 완비, 핵심 |
| 할 일 | **지금(full)** | items 데이터 완비 |
| 산출물·문서 | **지금(일부)** | 산출물 현행, 보고서·회의록은 모듈 후 |
| 메일·소통 | **지금(메일만)** | 메일 현행, 결재·쪽지·공지는 모듈 후(P3) |
| 자재·거래처 | **보류(P3)** | 데이터 소스 신설 전제 |
| 사람·팀 | **부분(P2)** | 미배정·연락처 지금, 부하·추세는 계정·집계 후 |

→ **지금 실활성 = 프로젝트 + 할 일 + (산출물·메일 일부) + (사람·팀 일부).**

### 6.3 전체 위젯 → 대분류 확정 매핑(객체축)

- **프로젝트:** 진행 과제 현황, 단계 게이트 대기, 산출물 진행률, **KPI 스트립**(과제 건강도 요약 — 여기 귀속), **최근 변경/활동 피드**(과제 전황)
- **할 일:** 오늘·지난 마감, 차단 목록, 내 담당 할 일, **마감 캘린더**(마감 중심 → 여기 귀속)
- **산출물·문서:** 최근 산출물, (P3) 보고서 즐겨찾기·회의록·표준문서·온보딩 체크리스트
- **메일·소통:** 최근 메일, 미분류 메일함, (P3) 결재 대기·쪽지/알림·공지 이력
- **자재·거래처(P3):** 발주/입고 예정, 재고 부족, BOM 변경, 거래처 현황, 구매 승인 대기
- **사람·팀:** 미배정 작업(현행), 연락처(현행) · (P2) 팀원별 부하·처리량 추세 · (P3) 요청 큐·분석 링크
- **(대분류 아님) 유틸:** 계산기·변환 도구 — 후순위, 어느 화면 안 도구로

### 6.4 반영 시 영향(참고)

객체축으로 가면 앱 상단 탭(현 NAV_LAYOUT 5그룹)도 **프로젝트·할 일·산출물·문서·메일·소통·자재·거래처·사람·팀**으로 재명명/재배치가 필요. 이건 IA 변경이라 별도 반영 단계(owner 승인 후). lexicon group_* 키도 같이 교체.

## 미해결/확인 필요

- 위젯 인스턴스를 **고유 1개**로 둘지(현행: id 유일) vs 같은 위젯 복수 배치 허용할지 — 후자는 모델 변경(instance id) 필요. 현재는 1개 고정.
- 계정·권한 시스템 도입 시점(P2b)이 "내 담당/팀원별 부하/결재" 위젯들의 선행 조건.
