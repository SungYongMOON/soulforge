# dev-erp P2b 설계 — 계정·권한 + 계정별 위젯 저장 (RBAC visible-but-locked)

- 작성: 2026-06-13, claude_fable-5. **설계·계약 초안만**(이 PC). 실 구현·마이그레이션·서버 기동 = 다른 PC.
- 목표: "개인 도구 → 팀 ERP" 전환의 척추. ① 계정/로그인 ② 계정별 대시보드 레이아웃 서버 저장(logout 내성) ③ RBAC = 보이지만 잠김.
- 근거: 현행 스키마(core_person id/name/role, meta key/value, event_log append-only), 서버 라우팅(node:http, qp/req.method 분기, send()), 클라 localStorage(`dev_erp_widgets` 등).
- 단계 원칙: localhost 파일럿(owner 단독) → 팀 읽기전용 → 쓰기개방. P2b는 "쓰기개방" 직전 필수 토대.

## 0. 보안 가드레일 (불변)

- 비밀번호 입력·계정 생성·권한 부여는 **owner(사람)가 직접**. 에이전트는 자동 입력·생성 금지.
- 비밀번호는 평문 저장 금지 → 해시(scrypt/bcrypt). 토큰은 httpOnly 쿠키.
- 사내 실데이터는 읽기전용·실값 미기록 원칙 유지. 계정 정보는 dev-erp 자체 DB(gitignore)에만.
- claude 레인은 claude/ 브랜치만, main push 금지.

## 1. 데이터 모델 (DDL 초안 — SQLite 파일럿)

```sql
-- 계정: core_person(사람)와 1:1 또는 N:1 연결. 로그인 주체.
CREATE TABLE IF NOT EXISTS core_account (
  id TEXT PRIMARY KEY,
  person_id TEXT REFERENCES core_person(id),   -- 담당자 매핑(내 담당 위젯 등)
  username TEXT NOT NULL UNIQUE,
  pw_hash TEXT NOT NULL,                        -- scrypt 등, 평문 금지
  status TEXT NOT NULL DEFAULT 'active',        -- active|disabled
  created_at TEXT NOT NULL
);

-- 세션: 서버측. 토큰=httpOnly 쿠키.
CREATE TABLE IF NOT EXISTS auth_session (
  token TEXT PRIMARY KEY,
  account_id TEXT NOT NULL REFERENCES core_account(id),
  created_at TEXT NOT NULL,
  expires_at TEXT NOT NULL
);

-- 권한 그룹(역할) + 역할별 자원 권한 (visible-but-locked 핵심)
CREATE TABLE IF NOT EXISTS rbac_role (
  id TEXT PRIMARY KEY, name TEXT NOT NULL
);
CREATE TABLE IF NOT EXISTS rbac_account_role (
  account_id TEXT NOT NULL REFERENCES core_account(id),
  role_id TEXT NOT NULL REFERENCES rbac_role(id),
  PRIMARY KEY (account_id, role_id)
);
-- resource = 'view:items' | 'mod:gates' | 'widget:teamload' | 'action:item.create' ...
-- visible: 메뉴/위젯이 보이는가  access: 실제 열람·실행 가능한가 (false면 보이되 잠김)
CREATE TABLE IF NOT EXISTS rbac_permission (
  role_id TEXT NOT NULL REFERENCES rbac_role(id),
  resource TEXT NOT NULL,
  visible INTEGER NOT NULL DEFAULT 1,
  access  INTEGER NOT NULL DEFAULT 1,
  PRIMARY KEY (role_id, resource)
);

-- 계정별 대시보드 레이아웃 (현 localStorage dev_erp_widgets 이전 대상)
CREATE TABLE IF NOT EXISTS user_dashboard_layout (
  account_id TEXT PRIMARY KEY REFERENCES core_account(id),
  layout_json TEXT NOT NULL,        -- [{id,x,y,w,h,c}] 그대로 (현 클라 모델과 동일)
  updated_at TEXT NOT NULL
);
```

설계 노트: 레이아웃은 **계정당 JSON blob 1행**(현 클라 배열을 그대로 직렬화)이 가장 단순·정합. 정규화(위젯당 1행)는 과설계 — 위젯 수가 작고 통째로 저장/로드라 blob 권장. navGroup·pins·mode 등 다른 개인설정도 같은 패턴으로 `user_pref(account_id, key, value)` 한 테이블에 흡수 가능(선택).

## 2. API 계약

| 메서드·경로 | 입력 | 출력 | 비고 |
|---|---|---|---|
| POST /api/auth/login | {username, password} | Set-Cookie(token); {account, roles, perms} | 비밀번호는 owner가 직접 입력. 실패 401 |
| POST /api/auth/logout | — | {ok} | 세션 삭제 |
| GET /api/me | (쿠키) | {account, person_id, roles, perms[]} | 미로그인 시 {anonymous:true} |
| GET /api/dashboard/layout | (쿠키) | {layout: [...]} 또는 기본 레이아웃 | 미로그인/없음 → DEFAULT_DASH |
| PUT /api/dashboard/layout | {layout:[...]} | {ok, updated_at} | 드래그/리사이즈/추가/삭제 후 호출(디바운스) |
| GET /api/rbac/roles (admin) | — | roles+permissions | 권한 편집 화면용 |
| PUT /api/rbac/permission (admin) | {role_id,resource,visible,access} | {ok} | owner/admin만 |

`perms[]` 형식 예: `[{resource:"mod:gates", visible:true, access:false}, ...]`. 클라는 이걸로 UI 게이팅.

## 3. RBAC = visible-but-locked (ECount 관찰 모델)

- 두 축 분리: **visible**(메뉴·위젯 노출) / **access**(열람·실행).
- `visible=true, access=false` → 메뉴/위젯은 보이되 클릭 시 "권한 없음" 잠김(자물쇠 아이콘). ECount 관찰과 동일.
- `visible=false` → 아예 숨김.
- 미정의 자원 = 기본 허용(파일럿) 또는 기본 차단(운영) — owner 정책 선택(아래 결정 필요).

권한 매트릭스 예시(초안):

| 역할 \ 자원 | 프로젝트·할일(view) | 자재·구매(mod) | 결재(action) | RBAC 편집(admin) |
|---|---|---|---|---|
| 팀장(owner) | 보임/가능 | 보임/가능 | 보임/가능 | 보임/가능 |
| 팀원 | 보임/가능 | 보임/잠김 | 보임/잠김 | 숨김 |
| 외부/읽기 | 보임/읽기 | 숨김 | 숨김 | 숨김 |

## 4. 클라이언트 변경(다른 PC 구현 시)

- 부팅: `GET /api/me` → state.account/perms. `GET /api/dashboard/layout`로 위젯 배치 로드(현 localStorage 대체).
- 저장: dashLayout 변경부(saveDashLayout)에서 `PUT /api/dashboard/layout`(디바운스 ~500ms). 미로그인 시 localStorage 폴백 유지.
- 게이팅: renderNav/위젯에서 perms 적용 — access=false면 비활성+자물쇠, visible=false면 미렌더.
- 마이그레이션: 첫 로그인 시 서버 레이아웃이 비었고 localStorage에 있으면 "기존 배치 가져오기?" 1회 제안 → PUT.

## 5. 단계(이 PC vs 다른 PC)

- **이 PC(설계 완료분)**: 위 DDL·API 계약·RBAC 매트릭스·클라 게이팅 지점.
- **다른 PC(실구현)**: 마이그레이션 SQL 적용, auth/세션·해시 구현, /api/* 라우트, 클라 적용, node:test(로그인·레이아웃 저장/로드·권한 게이팅), 브라우저 검증.
- **owner 결정 필요**: ① 미정의 자원 기본값(허용 vs 차단) ② 계정-사람 매핑 방식(1:1 강제 여부) ③ 세션 만료 시간.

## 6. 테스트 계획(다른 PC)

- 단위: 비밀번호 해시 검증, 권한 판정(visible/access 조합 4종), 레이아웃 저장→로드 라운드트립.
- 통합: 미로그인=기본 레이아웃·localStorage 폴백 / 로그인=계정 레이아웃 / 권한없는 모듈 잠김 표시 / 로그아웃 후 재로그인 레이아웃 유지(logout 내성).
- lexicon: 잠김/로그인/로그아웃 라벨 이중사전 parity.
